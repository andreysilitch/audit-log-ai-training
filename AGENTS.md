# AGENTS.md

## Project map

Внутренний audit-сервис для приёма, хранения и поиска аудит-событий компании.

Основные потребители данных: compliance-офицеры, SRE и security-аналитики. Сервис используется для compliance, security и observability, поэтому приоритеты проекта — корректность, неизменяемость записей, надёжность хранения и удобный поиск.

Ожидаемая технологическая база:

- Java 21
- Spring Boot 3
- Gradle Kotlin DSL
- PostgreSQL
- Flyway для миграций схемы БД
- Testcontainers для интеграционных тестов

Ключевая функциональность:

- `POST /audit-events` — приём одного аудит-события.
- Поиск событий по `actor`, `resource` и временному диапазону.
- Retention policy: хранение событий N дней, затем archival.
- Опционально: tamper-evidence через hash chain.

Основная модель события:

- `timestamp` — время события, устанавливается только сервером.
- `actor` — инициатор действия: user id или service account.
- `action` — действие, например `resource.updated`, `user.login`.
- `resource` — объект действия, например `project:42`, `invoice:777`.
- `outcome` — результат: `success`, `denied`, `error`.
- `context` — произвольный JSON с деталями.

## Invariants

Эти правила являются жёсткими и не должны нарушаться ни в API, ни в доменной логике, ни в миграциях, ни в тестах.

1. **Append-only storage**

   Аудит-события нельзя изменять или удалять после записи.

   Запрещено:

   - `UPDATE` существующих аудит-событий;
   - `DELETE` существующих аудит-событий;
   - API-эндпоинты для редактирования событий;
   - API-эндпоинты для удаления событий.

   Любое исправление или уточнение должно оформляться новым аудит-событием, а не изменением старого.

2. **Timestamp устанавливает только сервер**

   Клиент не должен передавать `timestamp` как источник истины. Если поле времени есть во входном DTO, оно не должно использоваться как `timestamp` события.

   Сервер обязан установить `timestamp` самостоятельно в момент приёма события или в строго определённой точке доменной логики.

3. **Actor обязателен**

   Поле `actor` обязательно для каждого события.

   Нельзя сохранять событие, если `actor`:

   - отсутствует;
   - пустой;
   - состоит только из пробелов.

4. **Событие должно быть пригодно для расследования**

   Каждое сохранённое событие должно содержать минимум:

   - `timestamp`;
   - `actor`;
   - `action`;
   - `resource`;
   - `outcome`.

   `context` может быть пустым, но должен храниться как JSON-структура, а не как невалидная строка.

5. **Retention не равен delete бизнес-данных вручную**

   Retention policy не должна превращаться в произвольное удаление аудит-записей. После истечения N дней события должны переходить в archival согласно явно заданному процессу.

   Если физическое удаление из основной таблицы всё же понадобится после архивации, это должно быть отдельным архитектурным решением с тестами, аудитом процесса и понятным подтверждением, что immutable-требование сохраняется в архивном хранилище.

6. **Tamper-evidence не должен быть декоративным**

   Если реализуется hash chain, он должен покрывать порядок событий и содержимое события. Нельзя добавлять поле hash без проверки цепочки и тестов на обнаружение изменения исторических данных.

## Architectural rules

1. **Доменная логика не должна зависеть от Spring MVC**

   Валидация инвариантов и создание события должны находиться в сервисном/доменном слое, а не только в контроллере.

   Контроллер отвечает за HTTP-контракт, сериализацию и коды ответа. Он не должен быть единственным местом, где защищаются invariants.

2. **API приёма событий не принимает серверные поля от клиента**

   `POST /audit-events` должен принимать только клиентские данные:

   - `actor`;
   - `action`;
   - `resource`;
   - `outcome`;
   - `context`.

   `timestamp`, технический `id`, hash-chain поля и archival-поля выставляются сервером.

3. **PostgreSQL-схема должна защищать invariants**

   Ограничения должны быть продублированы на уровне БД там, где это возможно:

   - `actor NOT NULL`;
   - `action NOT NULL`;
   - `resource NOT NULL`;
   - `outcome NOT NULL`;
   - `timestamp NOT NULL`;
   - `context` как `jsonb`;
   - индексы для поиска по `actor`, `resource`, `timestamp`.

   Приложение не должно полагаться только на Java-валидацию.

4. **Миграции только через Flyway**

   Все изменения схемы PostgreSQL должны проходить через Flyway-миграции.

   Запрещено:

   - ручное изменение схемы без миграции;
   - использование `ddl-auto=update` для production-like окружений;
   - изменение уже применённых миграций после их попадания в main.

5. **Поиск должен быть явно ограничен временным диапазоном**

   Поиск аудит-событий должен поддерживать фильтры по:

   - `actor`;
   - `resource`;
   - `time range`.

   Для больших таблиц нельзя проектировать поиск как полный scan без индексов и пагинации.

6. **Интеграционные тесты должны использовать Testcontainers**

   Логика, завязанная на PostgreSQL, Flyway, JSONB, индексируемые поля и транзакции, должна проверяться на реальном PostgreSQL через Testcontainers.

   H2 или in-memory базы не должны использоваться как замена PostgreSQL для интеграционных тестов persistence-слоя.

7. **Нет UPDATE/DELETE в repository API для audit events**

   Repository-интерфейс для аудит-событий должен предоставлять операции вида:

   - append/save new event;
   - find/search;
   - archive according to retention flow, если это явно спроектировано.

   Он не должен предоставлять публичные методы общего назначения `update`, `deleteById`, `deleteAll` для аудит-событий.

8. **Retention и archival должны быть отдельным модулем**

   Логику retention нельзя размазывать по контроллерам или repository.

   Она должна быть выделена в отдельный сервис, например `RetentionPolicyService`, с понятными правилами:

   - сколько дней хранить в основной таблице;
   - куда архивировать;
   - как подтверждать успешную архивацию;
   - что делать при ошибке архивации.

9. **Ошибки API должны быть безопасными**

   Ответы API не должны раскрывать чувствительные детали инфраструктуры, SQL-запросов, stack trace или внутренней схемы БД.

   При этом ошибки валидации должны быть достаточно понятными для сервисов-клиентов: например, `actor is required`, `outcome is invalid`.

10. **Context хранится как JSON, но не управляет схемой домена**

   `context` предназначен для дополнительных деталей. Основные поля события нельзя прятать только внутри `context`.

   Нельзя заменять `actor`, `action`, `resource`, `outcome` произвольными ключами внутри `context`.

11. **Hash chain, если реализуется, должен быть частью write path**

   Если включён tamper-evidence, hash предыдущего события и hash текущего события должны вычисляться при записи события.

   Нужно предусмотреть:

   - стабильную каноническую сериализацию события для hash;
   - защиту порядка событий;
   - тест, который ломает историческую запись и проверяет, что цепочка становится невалидной;
   - отдельную операцию проверки цепочки для security/compliance сценариев.

12. **События должны быть идемпотентно обрабатываемы на уровне клиентов или через отдельный ключ**

   Если сервисы-клиенты могут повторять запросы, нужно явно решить вопрос с дублями.

   По умолчанию аудит-событие считается новым фактом при каждом успешном `POST /audit-events`. Если нужна дедупликация, вводится отдельный client-provided idempotency key, но он не должен нарушать append-only модель.

13. **Наблюдаемость сервиса обязательна**

   Так как сервис нужен для observability и security, он сам должен быть наблюдаемым:

   - метрики успешных и ошибочных записей;
   - latency записи и поиска;
   - метрики archival/retention;
   - структурированные логи без утечки чувствительного `context`;
   - health checks для PostgreSQL.

14. **Безопасность доступа к чтению важнее удобства**

   Читать аудит-события должны только авторизованные роли: compliance-офицеры, SRE и security-аналитики.

   Поисковые API должны проектироваться с учётом контроля доступа, rate limits и защиты от массовой выгрузки без разрешения.
